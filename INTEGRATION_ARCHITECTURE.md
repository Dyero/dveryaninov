# Архитектура интеграции: 1С-Битрикс (Интернет-магазин) ↔ Битрикс24 CRM

Данный документ описывает потоки данных, сущности и требования к безопасности при обмене данными по REST API между интернет-магазином (на платформе 1С-Битрикс: Управление сайтом) и CRM Битрикс24.

**Специфика бизнеса:** Производство межкомнатных дверей.  
**Ключевая особенность:** Заказ из конфигуратора представляет собой не один монолитный товар, а сборный комплект (Бандл): Полотно + Погонаж + Фурнитура + Кастомные свойства (услуги врезки, нестандартные размеры).

---

## 🔒 Требования к безопасности и инфраструктуре

Поскольку сайт разрабатывается на платформе **«1С-Битрикс: Управление сайтом»** (БУС), а обмен идет с **Битрикс24**:
1. **Защита от SQL-инъекций:** Все запросы к БД на стороне сайта должны выполняться **исключительно через ядро D7 (ORM)** платформы 1С-Битрикс. Прямые запросы (`$DB->Query`) без подготовленных выражений и привязки параметров (prepared statements) строго запрещены.
2. **Санитизация данных:** Все входящие данные из форм (конфигуратор, регистрация) должны проходить строгую типизацию и экранирование `htmlspecialchars` при выводе, а также валидацию на стороне сервера перед отправкой в B24.
3. **Безопасность API:** 
   - Обмен данными должен осуществляться только по протоколу **HTTPS**.
   - Для обмена использовать **Локальное приложение Битрикс24** с OAuth 2.0 или безопасные **входящие/исходящие вебхуки** с ограничением прав (scope) только на необходимые сущности (crm, catalog, user).
   - Токены и доступы хранить в зашифрованном виде или в защищенных константах ядра, исключить их попадание в JS-код фронтенда.

## 1. Интеграция Каталога (Битрикс24 → Сайт)

Сайт должен регулярно получать актуальные данные из каталога Битрикс24 для синхронизации цен и остатков.
* **Механизм:** Настроенный cron-агент на стороне БУС, опрашивающий REST API Битрикс24 (метод `crm.product.list`), либо обработка исходящих вебхуков из B24 при изменении товара.
* **Передаваемые данные:**
  - **Товары (Полотна, Погонаж, Фурнитура):** `ID товара` в Б24, Название, Базовая цена, Акционная цена, Остаток, ID Категории.
  - **Торговые предложения (SKU):** Вариации для полотен (набор стандартных размеров, например, 2000х800, и стандартных цветов, например, RAL 9003). В БУС они ложатся в инфоблок торговых предложений.
  - **Свойства (Характеристики):** Коллекция, Стиль, Наполнение, Покрытие, Наличие стекла и т.д.

## 2. Синхронизация Пользователей / Личный кабинет (Двусторонний обмен)

Каждый клиент на сайте (сущность "Пользователь" в БУС) жестко связан с сущностью «Контакт» (`crm.contact`) в Битрикс24.

* **Поток: Сайт → Битрикс24 (Регистрация / Обновление)**
  - Данные: Имя, Фамилия, E-mail, Телефон (ключевой идентификатор), Адрес доставки (в реквизиты).
  - Логика поиска дублей: Сайт вызывает `crm.contact.list` с фильтром по номеру телефона (с предварительной очисткой от спецсимволов).
  - Действие: Если контакт найден — сайт сохраняет его `CONTACT_ID` в пользовательское поле (UF_CRM_ID) профиля пользователя сайта. Если нет — вызывает `crm.contact.add` и сохраняет полученный `CONTACT_ID`.

* **Поток: Битрикс24 → Сайт (Авторизация / Обновление сессии)**
  - Действие: Проверка статуса контакта, загрузка персональной скидки клиента (если в CRM ведется программа лояльности), обновление данных профиля при входе в ЛК.

## 3. Передача Заказа из Конфигуратора (Сайт → Битрикс24) ⚠️ КРИТИЧЕСКИЙ БЛОК

> **ВНИМАНИЕ РАЗРАБОТЧИКАМ:** Заказ одной скомплектованной двери **КАТЕГОРИЧЕСКИ ЗАПРЕЩАЕТСЯ** передавать в CRM как одну монолитную товарную позицию с кучей комментариев! Это сломает складской учет фурнитуры и погонажа, а сметы превратятся в кашу. Заказ должен раскладываться на отдельные товарные позиции (Line Items).

При оформлении заказа сайт создает **Сделку** (`crm.deal.add`) и привязывает к ней **Товарные позиции** (`crm.deal.productrows.set`).

### 3.1. Создание Сделки (Мастер-данные)
- `CONTACT_ID`: Привязка к клиенту из шага 2.
- `STAGE_ID`: Стадия ("Новая заявка с сайта").
- `SOURCE_ID`: Источник ("Сайт").
- `OPPORTUNITY`: Итоговая сумма (рассчитывается CRM автоматически на основе productrows).
- **Пользовательские поля (UF_*):**
  - Комментарий клиента.
  - Нужна ли доставка (Да/Нет).
  - Нужна ли установка (Да/Нет).
  - Примененный промокод.

### 3.2. Товарные позиции (Line Items внутри Сделки)
Массив товаров, который передает сайт за **одну** собранную дверь. 
Для группировки комплектов вводится специальное пользовательское поле в строке товара (или комментарий к строке) — **Идентификатор комплекта (`Bundle_ID`)**. 
Пример для 5 дверей в корзине: все запчасти первой двери имеют `Bundle_ID = 1`, второй — `Bundle_ID = 2`. Это позволит менеджеру понять, какие ручки идут к какому полотну.

**Структура массива для одной двери (`Bundle_ID = 1`):**
1. **Полотно:** ID товара, Количество (1 шт), Цена.
   - *Комментарий к позиции:* Сторона открывания (Левая/Правая), Тип (Распашная). Если "Свой размер" — точные Высота (мм) и Ширина (мм).
2. **Погонаж (Короб):** ID товара, Количество (2.5 шт или 3 шт в зависимости от логики расчета), Цена.
3. **Погонаж (Наличники):** ID товара, Количество (5 шт), Цена.
4. **Фурнитура (Ручка):** ID товара, Количество (1 шт), Цена.
5. **Фурнитура (Замок):** ID товара, Количество (1 шт), Цена.
6. **Доп. услуги:** ID услуги (например, товар типа "услуга" — Врезка фурнитуры), Количество (1 шт), Цена.

## 4. История заказов для Личного кабинета (Битрикс24 → Сайт)

В разделе «Мои заказы» сайт запрашивает актуальные статусы сделок клиента в реальном времени.

* **Запрос:** `crm.deal.list` с фильтром `CONTACT_ID = текущий_пользователь.UF_CRM_ID`.
* **Передаваемые данные (JSON):**
  - Номер заказа (ID Сделки).
  - Дата создания (`DATE_CREATE`).
  - Итоговая сумма (`OPPORTUNITY`).
* **Статус заказа (Воронка):**
  - Идентификатор стадии (`STAGE_ID`) мапится на стороне сайта в понятные клиенту статусы (например: `PREPARATION` -> "В обработке", `EXECUTING` -> "На производстве", `WON` -> "Выполнено").
* **Состав заказа:**
  - Сайт дополнительно запрашивает `crm.deal.productrows.get` для вывода списка купленных товаров в заказе.
  - На основе этих данных формируется кнопка **"Повторить заказ"** (добавляет эти же ID товаров в корзину сайта, что критически важно для B2B-клиентов: дизайнеров и прорабов).
