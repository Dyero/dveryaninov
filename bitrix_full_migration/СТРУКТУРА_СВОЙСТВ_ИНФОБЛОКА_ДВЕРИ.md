# Структура свойств инфоблока "Дверные полотна" (ID: 6)

## Таблица свойств

| Код свойства        | Название                    | Тип                      | Множественное | Обязательное | Примечание |
|--------------------|-----------------------------|--------------------------|---------------|--------------|------------|
| COLLECTION          | Коллекция                   | Строка                   | Нет           | Да           | Название коллекции (Альберта, Флай, и т.д.) |

| BLADE_TYPE          | Тип полотна                 | Список                   | **Да**        | Да           | Значения: "ПГ" (полотно глухое), "ПО" (полотно остеклённое) |

| MOLDING_SUPPORT     | Поддержка молдингов         | Да/Нет                   | Нет           | Нет          | Возможность добавления молдингов |
| IMAGE               | Изображение двери           | Файл                     | Нет           | Нет          | Основное изображение двери |

## Важные изменения

### 1. Тип полотна (BLADE_TYPE)
- **Тип изменён на:** Список с множественным выбором
- **Значения списка:**
  - `PG` → "ПГ" (Полотно глухое)
  - `PO` → "ПО" (Полотно остеклённое)
- **Множественное:** ДА (галочка установлена)
- **Причина:** Многие двери доступны в обоих вариантах (и ПГ, и ПО)

### 2. Ценообразование

#### Проблема
У одной модели двери могут быть разные цены для ПГ и ПО.

#### Решение
Используются два отдельных свойства:
- **BASE_PRICE_PG** — цена для глухого полотна
- **BASE_PRICE_PO** — цена для остеклённого полотна

#### Логика работы в коде:

```php
// Получение элемента
$arProduct = CIBlockElement::GetList(
    [],
    ["ID" => $productId],
    false,
    false,
    ["ID", "NAME", "PROPERTY_BLADE_TYPE", "PROPERTY_BASE_PRICE_PG", "PROPERTY_BASE_PRICE_PO"]
)->GetNext();

// Получаем доступные типы полотна (может быть массив)
$bladeTypes = $arProduct["PROPERTY_BLADE_TYPE_VALUE"]; // Вернёт массив, например ["ПГ", "ПО"]

// Определяем цену в зависимости от выбранного типа
$selectedType = $_REQUEST['blade_type'] ?? 'PG'; // Выбор пользователя

if ($selectedType === 'PG' || $selectedType === 'ПГ') {
    $price = $arProduct["PROPERTY_BASE_PRICE_PG_VALUE"];
} elseif ($selectedType === 'PO' || $selectedType === 'ПО') {
    $price = $arProduct["PROPERTY_BASE_PRICE_PO_VALUE"];
}

// Вывод доступных вариантов
echo "Доступные типы: ";
if (is_array($bladeTypes)) {
    foreach ($bladeTypes as $type) {
        $priceField = ($type === 'ПГ') ? 'BASE_PRICE_PG' : 'BASE_PRICE_PO';
        $typePrice = $arProduct["PROPERTY_" . $priceField . "_VALUE"];
        echo "$type - " . number_format($typePrice, 0, ',', ' ') . " ₽; ";
    }
}
```

## Примеры использования

### Пример 1: Дверь доступна только в ПГ
```
COLLECTION: Альберта
MODEL_NUMBER: 1
BLADE_TYPE: [ПГ]
BASE_PRICE_PG: 23140
BASE_PRICE_PO: (пусто)
```

### Пример 2: Дверь доступна в обоих вариантах
```
COLLECTION: Флай
MODEL_NUMBER: 8
BLADE_TYPE: [ПГ, ПО]
BASE_PRICE_PG: 52800
BASE_PRICE_PO: 58600
```

### Пример 3: Дверь доступна только в ПО
```
COLLECTION: Аврора
MODEL_NUMBER: 3
BLADE_TYPE: [ПО]
BASE_PRICE_PG: (пусто)
BASE_PRICE_PO: 23140
```

## Настройка в админ-панели Bitrix

### Шаг 1: Создание списка для BLADE_TYPE
1. Перейти в **Контент → Структура инфоблоков**
2. Выбрать инфоблок "Дверные полотна" (ID: 6)
3. Перейти на вкладку **Свойства**
4. Найти свойство **BLADE_TYPE** (или создать новое)
5. Настроить:
   - **Код:** `BLADE_TYPE`
   - **Название:** Тип полотна
   - **Тип:** Список
   - **Множественное:** ✅ ДА
   - **Обязательное:** ✅ ДА
   - **Значения списка:**
     - ID: `PG`, Значение: `ПГ`, Сортировка: 100
     - ID: `PO`, Значение: `ПО`, Сортировка: 200

### Шаг 2: Создание свойств цен
Создать два числовых свойства:

**Свойство 1:**
- Код: `BASE_PRICE_PG`
- Название: Базовая цена ПГ
- Тип: Число
- Множественное: Нет

**Свойство 2:**
- Код: `BASE_PRICE_PO`
- Название: Базовая цена ПО
- Тип: Число
- Множественное: Нет

## Компонент для вывода каталога

Компонент `dveryaninov:doors.catalog` должен учитывать эту структуру и:
1. Выводить корректную цену в зависимости от выбранного типа полотна
2. Показывать оба варианта, если дверь доступна и в ПГ, и в ПО
3. Фильтровать по типу полотна через параметры компонента

## Миграция существующих данных

Если в инфоблоке уже есть товары со старой структурой:

1. **Старая структура:** Одно поле `PRICE` и одно поле `VARIANT` (строка "ПГ / ПО")
2. **Новая структура:** Массив `BLADE_TYPE` и два поля `BASE_PRICE_PG`, `BASE_PRICE_PO`

### Скрипт миграции (выполнить один раз):

```php
<?php
require($_SERVER["DOCUMENT_ROOT"]."/bitrix/modules/main/include/prolog_before.php");

CModule::IncludeModule("iblock");

$rsElements = CIBlockElement::GetList(
    [],
    ["IBLOCK_ID" => 6],
    false,
    false,
    ["ID", "NAME", "PROPERTY_PRICE", "PROPERTY_VARIANT"]
);

while ($arElement = $rsElements->GetNext()) {
    $oldVariant = $arElement["PROPERTY_VARIANT_VALUE"] ?? "";
    $oldPrice = $arElement["PROPERTY_PRICE_VALUE"] ?? 0;

    $newBladeTypes = [];
    $pricePG = null;
    $pricePO = null;

    // Определяем доступные типы из старого поля VARIANT
    if (strpos($oldVariant, "ПГ") !== false) {
        $newBladeTypes[] = "PG";
        $pricePG = $oldPrice;
    }
    if (strpos($oldVariant, "ПО") !== false) {
        $newBladeTypes[] = "PO";
        $pricePO = $oldPrice;
    }

    // Обновляем элемент
    CIBlockElement::SetPropertyValuesEx($arElement["ID"], 6, [
        "BLADE_TYPE" => $newBladeTypes,
        "BASE_PRICE_PG" => $pricePG,
        "BASE_PRICE_PO" => $pricePO
    ]);

    echo "Обновлён элемент ID: " . $arElement["ID"] . " - " . $arElement["NAME"] . "\n";
}

echo "Миграция завершена!";
?>
```

## Связь с погонажом (ID: 5)

Инфоблок погонажа (молдинги, наличники, и т.д.) имеет отдельную структуру и не связан напрямую с типом полотна двери. Связь осуществляется через свойство **MOLDING_SUPPORT** в инфоблоке дверей.
